## JSP와 서블릿, 스프링에서 발생할 수 있는 문제점

아무리 잘 짜여진 자바 기반의 웹 프레임워키를 사용 한다고 해도, 
<br>기본적인 JSP와 Servlet의 개념은 알고 있어야한다.

JSP의 라이프 사이클은 다음의 단계를 거친다.
1. JSP URL 호출
2. 페이지 번역
3. JSP 페이지 컴파일
4. 클래스 로드
5. 인스턴스 생성
6. jspInit 메서드 호출
7. _jspService 호출
8. jspDestory 호출

JSP의 include 기능에 대해서도 알아보자.

- 정적인 방식 : <%@include file = "관련URL"%>
- 동적인 방식 : <jsp:include page = "!~!~!">

동적인 방식이 정적인 방식보다 느리다. 약 30배나 느리다.
<br>성능을 빠르게 하려면 정적인 방식을 사용해야한다는 의미가 된다. 하지만 모든 화면을 정적인 방식으로 활용하면 오류가 발생할 수 있다. 그러므로 상황에 맞는 include방식을 사용하자.

### 자바 빈즈, 잘 쓰면 약, 잘못쓰면 독
자바 빈즈는 UI에서 서버측에 데이터를 담아서 처리하기 위한 컴포넌트이다.
<br>간단히 자바 빈즈의 문제점만 알아보자.

자바빈즈를 통해 많은 성능에 영향을 미치진 않지만, jsp에서 소요되는 시간이 증가될 수 있다.

### 스프링 프레임워크 
개발된지 10년이 넘었다. POJO에 대한 설명과 IOC/DI등에 대한 설명이 적혀있다.
<br>이 부분에 대해서 적을까 했지만, 알고있는 내용이고, 이번주 CS스터디에 따로 기제할 내용이라 적지않는다.

스프링을 사용하면서 가장 많이 겪는 문제들은, 프록시 부분이다. 프록시는 기본적으로 실행 시에 생성된다. 따라서 개발할 때 적은 요청을 할 때는 이상없다가 요청량이 많으면 문제가 될 수 있다.

주요 기능은 @Transactional 어노테이션을 사용하면 해당 어노테이션을 사용한 클래스의 인스턴스 만들때 프록시 객체를 만든다.

스프링 자체에서 제공하는 기능은 많은 검증을 받고 했지만, 개발자가 직접 개발한 AOP에서 많은 문제가 터진다.

추가로 스프링 내부 매커니즘에서 사용하는 캐시도 조심해서 써야한다. 스프링 MVC에서 작성하는 메서드의 리턴 타입으로 다음과 같은 문자열을 사용할 수 있다. 실제 뷰 객체를 찾는 매커니즘을 사용하는데, 이때 매번 동일한 문자에 대해 찾기보다는 이미 찾아본 뷰 객체를 캐싱해두면 다음에도 동일한 문자열이 반환됐을때 더 쉽게 찾을 수 있다.

스프링에서 제공하는 뷰 리졸버중에 자주사용되는 InternalResourceViewResovler에는 그러한 캐싱 기능이 내장되어있다.

매번 다른 문자열이 생성될 가능성이 높고 상당히 많은 수의 키 값으로 캐시 값이 생성될 여지가 있는 상황에서는 메모리에 치명적일 수 있다.

프레임워크에 대한 제대로된 이해 없이 선배들이 넘겨준 코드를 대충 복사하여 사용하면 큰 장애가 생긴다.

각 어노테이션이 어떤 의미인지는 정확히 알고 사용해야한다.
