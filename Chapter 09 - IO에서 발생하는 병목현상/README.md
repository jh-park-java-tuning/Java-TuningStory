## IO에서 발생하는 병목현상

이 부분도 잘못 사용하면, 응답속도에 많은 영향을 준다.
<br>이번 내용은 자바 IO와 NIO의 파일 처리 시 성능상 비교 결과를 보면서 얘기해보겠다.

NIO는 Non Blocking IO라고 해서 자바 1.4에 나온것이다. 그걸 쓰면 자바도 IO가 빠르다. 라고 했지만 알고보니 New~ 이다.

### 기본적인 IO처리는 이렇게 한다.
자바에서 입력과 출력은 스트림을 통해서 이루어진다. 어떤 디바이스를 통해 이뤄지는 작업을 모두 IO라고 한다.

System.out.print에서 out은 PrintStream을 System클래스에서 static으로 정해놓은 변수이다.

참고로 IO는 성능에 가장 많은 영향을 끼친다. IO에서 발생하는 시간은 CPU를 사용하는 시간과 대기 시간에 속하기 때문이다.

개발된 애플리케이션에서 IO를 사용했을때 IO와 관련된 디바이스가 느리면 느릴수록 애플리케이션의 속도는 느려진다.

이후에 Stream에 대해서 소개하고 버퍼에대해서 소개하는데 적지는 않겠다.

바이트나, 문자열 단위로 읽을때 반드시 닫아주어야한다. 리소스가 부족해진다.
<br>이후에 readCharStream 메서드에 대한 소스코드가 나온다.

결과는 윈도우 기반 10MB정도의 파일을 읽는 속도는 2480초가 나왔다.
이것은 문자열을 하나씩 다 읽었기때문이다.

버퍼의 크기가 8개이고, 마지막에 읽은 값이 네자리라고하자. 이때 매번 초기화 하지않으면 마지막에 들어가는 값은 GOODO!!! 가 된다(예시 생략)
<br>우리가 필요한 부분은 GOOD이므로, 이 부분에 대한 처리를 해줘야한다.

### NIO의 원리는 어떻게 될까?
알기전에 근본적으로 IO작업이 운영체제에서 어떻게 수행되는지 알아야한다.

자바를 사용하여 하드디스크에 있는 데이터를 읽는다면 어떻게 진행될까?
1. 파일을 읽으라는 메서드를 자바에 전달한다.
2. 파일명을 전달받은 메서드가 운영체제의 커널에게 파일을 읽어달라고 요청한다.
3. 커널이 하드 디스크로부터 파일을 읽어서 자신의 커널에 있는 버퍼에 복사하는 작업을 수행한다. DMA에서 이 작업을 하게된다.
4. 자바에서는 마음대로 커널의 버퍼를 사용하지 못하므로 JVM으로 그 데이터를 전달한다.
5. JVM에서 메서드에 있는 스트림 관리 클래스를 사용하여 데이터를 처리한다.

자바에서는 3번이나 4번을 할 때 대기시간이 발생할 수 밖에 없다. 이러한 단점을 보완하기 위해 NIO가 나온것이다. 

- 버퍼의 도입
- 채널의 도입
- 문자열의 엔코더와 디코더 제공
- Perl 스타일의 정규 표현식에 기초한 패턴 매칭 방법 제공
- 파일을 잠그거나 메모리 매핑이 가능한 파일 인터페이스 제공
- 서버를 위한 복합적인 Non-blocking IO 제공

### lastModified() 메서드의 성능 저하
자바에서 파일이 변경되었는지 확인하기 위해서 File클래스에 있는 lastModified()라는 메서드를 사용해왔다. 이 메서드를 사용하면 최종 수정된 시간을 밀리초단위로 제공한다.

JMH를 사용했을 경우, 58마이크로초가 나왔다. 소스코드는 생략했다.

이렇게 보면 큰 문제가 되지않지만, 반복하는 형태의 서비스를 제공하면 말이 달라진다. 이작업은 IO작업을 수반하기 때문에 OS의 IO의 영향을 많이 받을 수 밖에 없다.

JDK 7부터 새로운 개념의 IO를 처리한다. WatcherService에 대해 알아보자.

(일단 책으로 다 봤다. 소스코드가 길어서 생략한다.)

JDK 7 이상부터는 파일 구조 변경을 확인할 필요가 없어졌다. Watch관련 클래스만 활용해도 파일을 쉽게 모니터링할 수 있기 때문이다.
