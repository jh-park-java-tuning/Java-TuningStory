## 왜 자꾸 String을 쓰지 말라는거야

- 모든 개발자가 알면서도 지키지 않는것이 있다. 바로 문자열 관련 부분이다.
- String은 잘 사용하면 상관없지만, 잘못 사용하면 메모리에 많은 영향을 준다.

> GC는 Garbage Collection 의 약자로, 자바에서 사용하는 한정된 메모리 공간이 꽉 찼을때
> <br> 더이상 필요 없는 객체들을 제거하는 작업을 의미한다. 이에 대한 내용은 17에 나와있다.

- String이 GC에 영향을 주는 것은 확실하다.

### String 클래스를 잘못 사용한 사례
- 자바 기반의 프로그래밍을 할 때 java.lang.Object를 제외하고 가장 많이 사용하는 객체가 뭘까?
- 기본 자료형을 제외하면 가장 많이 사용하는 객체는 String, 2위는 Collection 관련 클래스라고 생각한다.
- 대부분의 웹 기반 시스템은 DB에서 가지고 와서 데이터를 화면에 출력하는 시스템이어서
<br> 쿼리 문장을 만들기 위한 String클래스와 결과를 처리하기 위한 Collection 클래스가 가장 많이 사용된다.

```java
String strSQL = "";
strSQL += "select * ";
strSQL += "from ( ";
strSQL += "select A_column, ";
strSQL += "B-column, ";
```
요즘은 myBatis, Hibernate같은 데이터 매핑 프레임 워크를 사용하지만 예전에는 이렇게 사용했다.
문자열을 더하면 100회 수행하면 총 400라인이 된다는 가정하에 몇 MB의 메모리를 사용하는지 직접 확인하기로 했다.

- 메모리 사용량 10회 평균 약 5MB 가 나왔다.
- 응답시간 10회 평균 약 5ms가 나왔다.

### StringBuffer 클래스와 StringBuilder 클래스
- JDK 5.0 기준으로 문자열을 만드는 클래스는 버퍼, 빌더가 가장 많이 사용된다.
- 그러면 StringBuffer 클래스와 StringBuilder 클래스는 뭐가 다를까?
- StringBuffer 클래스는 스레드에 안전하게 설계되어있다. <br>
여러개의 스레드에서 하나의 버퍼를 처리해도 전혀 문제가 되지않는다.
- StringBuilder는 단일 스레드에서 안전성만 보장한다. 그렇게에 여러개의 스레드에서 오류가 난다.

> 1. String Buffer() : 아무 값도 없는 StringBuffer 객체를 생성한다. 기본 용량은 16개의 char이다.
> 2. StringBuffer(CharSequence seq) : CharSequence를 매개변수로 받아 그 seq값을 갖는 StringBuffer를 생성한다.
> 3. StringBuffer(int capacity) : capacity에 지정한 만큼의 용량을 갖는 StringBuffer를 생성한다.
> 4. StringBuffer(String str) : str값을 갖는 StringBuffer를 생성한다.

**CharSequence seq**는 뭘까? 이것은 인터페이스다. 다시 말하면 클래스가 아니어서
<br>인스턴스로 객체를 생성할 수 없다.

<br> 메모리는 3390배 더 사용한다 String이 훨씬,

왜 문제가 발생할까? 

- a에 Value를 더하면 새로운 String 클래스에 객체가 만들어지고 기존에 있던 a객체는 GC의 대상이 된다. <br>
- String이 새로운 객체를 만드는 작업을 수행할때마다 새로운 주소와 새로운 객체가 만들어진다,
- 즉 계속해서 쓰레기값인 GC 대상이 된다.

프로그래밍을 하면서 최적화나, 메모리 사용 최소화는 당연한것이다.

- StringBuffer와 StringBuilder는 새로운 객체를 생성하지않는다.

### 그렇다면 String은 무조건 나쁜걸까?
- 아니다. 짧은 문자열을 더할 경우 사용된다.
- Buffer는 스레드에 안전한 프로그램이 필요할 때나 개발중인 시스템의 부분이 스레드에 안전한지 모를 경우 사용하면 좋다.
- Builder는 스레드의 안전 여부와 전혀 관계없는 프로그램을 개발할 때 좋다.

String 사용이 스레드에 안전한지는 6장 8장에서 설명한다.
